8. Règles de robustesse (16/16)

  • ✅ Contrats Pydantic — tous les agents ont des modèles d'entrée/sortie validés
  • ✅ Superviseur (T00) — consentement obligatoire, bornes de crawl, rate limiting
  • ✅ Mode consentement — affichage des limites avant crawl
  • ✅ Défensif uniquement — pas de test d'intrusion, pas de scanning de dossiers
  • ✅ Confidence scoring — chaque constat a un niveau High/Medium/Low
  • ✅ Source de preuve — chaque issue a agent source + URL + donnée observée + règle
  • ✅ do_not_recommend_if — pas de recommandation absurde (ajouter 300 mots à une page CGU)
  • ✅ Adaptatif — seuils par type de page (produit 300 mots, article 600, légal pas de seuil)
  • ✅ CMS-aware — recommandations localisées (WordPress→Yoast, PrestaShop→Module SEO)
  • ✅ Fallback — PSI→heuristique, GSC→estimation, Louvain→skip
  • ✅ Auto-disable — KE se désactive après 402, PSI après 429
  • ✅ Déduplication — T15 consolide les issues par URL+catégorie
  • ✅ Export multiple — HTML, JSON, CSV, PDF (weasyprint)
  • ✅ Routage inter-pipelines — T18 prépare les données pour P2/P4/P5/P6/P7
  • ✅ Rate limiting — configurable, respect robots.txt
  • ✅ Skip intelligent — T20 (logs) skippé automatiquement si pas de données

9. Méthode de travail

  1. L'utilisateur définit le process (quel pipeline, quels agents, quel ordre)
  2. L'utilisateur écrit le cahier des charges
  3. Claude implémente agent par agent, dans l'ordre défini
  4. Claude fait les tests (entrée valide, invalide, sortie conforme, erreur)
  5. Claude connecte au pipeline existant sans casser ce qui marche
  6. L'utilisateur valide chaque agent ou groupe d'agents
  7. L'utilisateur teste en conditions réelles
